# obsidian-raw-save

这是一个用于“确认稿原文保存”的 Skill。

它和现有的 `obsidian-summary-save` 不同：

- `obsidian-summary-save` 的职责是根据当前对话生成总结，再保存到 Obsidian
- `obsidian-raw-save` 的职责是把上一条 assistant 回复当作确认后的最终版本保存到 Obsidian

这个 Skill 适合下面这种工作流：

1. 你先和 AR 反复打磨总结内容
2. 你确认某一版已经满意
3. 你对 CLI 说 `原文保存` 或 `原文保存：<主题>`
4. CLI 读取上一条 assistant 回复，并将这一版内容保存到 Obsidian

## 触发词

- `原文保存`
- `原文保存：<主题>`

只有在你明确说出以上触发词时，才会启用这个 Skill。

## 核心行为

- 保存来源固定为“上一条 assistant 回复”
- 如果你提供主题，则用该主题作为文件名主题
- 如果你没有提供主题，则自动生成一个简短且具体的主题
- 每次都新建文件
- 不追加旧文件，也不覆盖已有文件

## 正文保存规则

- 目标是尽可能原样保存确认稿内容
- 如果上一条 assistant 回复已经是 Markdown，则直接保存
- 如果上一条 assistant 回复不是 Markdown，则先整理为 Markdown 再保存
- 这种整理只允许修改表现形式，不允许修改信息内容

允许的整理包括：

- 拆分段落
- 将列表整理为 Markdown 列表
- 为代码片段补 fenced code block
- 将明显的标题整理为 Markdown 标题
- 其他仅影响展示、不改变原意的 Markdown 格式整理

禁止的处理包括：

- 改写原句
- 重新总结
- 扩写或缩写内容
- 添加背景、结论、风险、待办等原文没有的章节
- 自动添加 YAML frontmatter
- 自动添加时间戳
- 在正文原本没有标题时，额外插入标题

## 保存位置

- 根目录：`C:\Users\fengx\onedrive\Apps\remotely-save\feng\00_Inbox\电脑\`
- 优先保存到当天日期目录，格式为 `YYYY-MM-DD`
- 如果当天目录不存在，则先创建

## 失败处理

以下情况不应直接保存，而应先向用户确认：

- 没有上一条 assistant 回复可用
- 上一条回复明显不是最终正文
- 上一条回复只是简短确认、过渡说明或建议性话术

## 使用示例

### 示例 1：直接保存确认稿

你已经和 AR 打磨完一版总结，上一条 assistant 回复就是最终稿。

你输入：

```text
原文保存
```

效果：

- Skill 读取上一条 assistant 回复
- 自动生成一个简短主题
- 将正文保存到当天目录下的新文件中

### 示例 2：指定主题保存

你输入：

```text
原文保存：AR 总结定稿
```

效果：

- 文件名使用 `AR 总结定稿` 作为主题
- 正文仍然取上一条 assistant 回复
- 每次新建，不污染旧文件

## 设计目标

这个 Skill 的目标不是“再生成一版更好的总结”，而是“忠实保存你已经确认满意的那一版内容”。
